2 Reflective Ability - Stepping Back

2.1 What I Would Do Differently

Start with database-native features

On LOCALTRIAGE, I initially used the rank-bm25 Python library for sparse retrieval, only to find it was too slow. Switching to PostgreSQL full-text search cost me a full day. Lesson: evaluate library choices against your production constraints before committing.

Build evaluation datasets early

I hit an annotation bottleneck, I needed labeled data faster than I could produce it. I had to reduce my evaluation set from 500 to 200 samples, widening confidence intervals. Next time: start annotation as a parallel workstream from day 1, and consider synthetic data generation.

Define "done" precisely to resist scope creep

Dashboard feature requests kept expanding during LOCALTRIAGE ("Can we add agent performance metrics?"). I learned to set strict MVP scope with explicit phase gates. This discipline also improved my project management in subsequent projects.

Think about async processing from the start

Retrofitting asynchronous pattern in a synchronous pipeline is painful. Building async into the architecture from step one would have provided a better UX and simplified scaling.

2.2 Methodology Changes I've Already Adopted

Baseline-first approach

After LOCALTRIAGE, I systematically quantify baselines before building complex solutions. On the HR Analytics project, this meant establishing a majority-class baseline (8 % recall) before engineering features. It makes improvement claims defensible and gives a fallback at every stage.

Modular architecture as a default

Every project now follows a component-swappable design. On LOCALTRIAGE, I could swap Qwen2.5-14B for 7B or 32B without touching pipeline code. On the MLOps project, the same separation let me swap scoring models while keeping the serving layer stable.

Evaluation is not optional

I pushed myself to build formal evaluation harnesses (RAGAS, F1 rubrics, confusion matrices) before tuning models. "Rubrics beat vibes" is now a personal motto. This made a tangible difference on the Semi-Supervised MRI project where threshold calibration was critical.

2.3 Progress Already Visible

  • Prompt engineering has high ROI: hours of prompt work saved weeks of potential fine-tuning on LOCALTRIAGE (73.3% routing accuracy without any task-specific training).
  • Hybrid retrieval is robust: combining dense and sparse consistently beats either alone, a pattern I've now replicated across 3 projects.
  • Documentation is part of the work: my recent projects ship with full BRDs, architecture docs, decision matrices, and retrospectives. This was not the case a year ago.
  • Testing discipline: from 0 tests in early projects to 84 % coverage (LOCALTRIAGE) and 195 tests (Planar).

2.4 What This Training Has Brought Me

This training fundamentally shifted my approach from "build first, evaluate later" to "measure first, build with evidence." Coming from a software engineering and security background, I had strong coding skills but lacked the rigorous evaluation methodology that defines production ML. The training gave me:

  • A structured ML lifecycle: problem framing → data audit → baseline → experimentation → deployment → monitoring. Before, I would jump to modeling too quickly.
  • Statistical rigor: understanding confidence intervals, cross-validation strategies, and the importance of proper train/val/test splits.
  • Business translation: the ability to convert model metrics (F1, recall) into business KPIs ($500 K savings, 64 % detection rate).
  • The MLOps mindset: a model is not done when it's trained, it's done when it's serving, monitored, and maintainable.

2.5 How My Perception of the ML Data Scientist Role Has Evolved

Before the training

I saw ML Data Scientist as primarily a modeling role, someone who trains models, tunes hyperparameters, and achieves the highest accuracy. The work ended when the notebook produced good metrics.

After the training and projects

I now understand the role as a full-stack data problem solver. The model is one component in a larger system that includes:

  • Data quality governance: garbage in, garbage out applies ruthlessly (KB quality limited LOCALTRIAGE more than model quality).
  • Stakeholder communication: agents want explanations, not just predictions. "Why this category?" matters more than raw accuracy.
  • Production engineering: latency budgets, memory management, containerization, monitoring, and drift detection are as important as model architecture.
  • Ethical responsibility: especially around data privacy (LOCALTRIAGE exists because cloud AI posed unacceptable privacy risks for sensitive data).
  • Continuous learning loops: feedback mechanisms that let the system improve from corrections, not just static training data.

The biggest shift: I now think of model deployment as the beginning of the ML lifecycle, not the end.
